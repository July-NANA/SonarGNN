# SonarGNN: 6人团队详细分工与协作流程方案

## 👥 6人详细分工与依赖说明 (全员端到端)

### **成员 A：数据构建与背景探索** ： FanYuhang
*   **代码与实验任务**：编写 `dataset.py`，从 `data/` 加载 CSV，构建 PyG 的 `Data` 对象。跑出数据类别分布饼图 (Licit vs Illicit)。
*   **报告与答辩任务**：撰写报告的 *Introduction* 和 *Dataset Description*。答辩时介绍项目背景、数据集概况及不平衡痛点。
*   **执行状态**：
    *   🟢 **可直接执行**：作为整个项目的起点，代码编写和报告的背景部分可以立即开工，不依赖任何人。
    *   🔴 **阻塞他人**：**必须最先完成 `dataset.py`**，否则成员 B、E 无法拿到数据对象，成员 F 无法测试评估指标。

### **成员 B：基线模型与基础训练** (Zhang Dingbang)
*   **代码与实验任务**：编写 `model.py` (GCN定义) 和基础 `train.py`。跑出基础 Epoch vs Loss/Acc 曲线图。
*   **报告与答辩任务**：撰写报告的 *Design & Functions* (网络架构设计)。答辩时介绍 GCN 架构，展示 Baseline 表现，引出性能痛点。
*   **执行状态**：
    *   🔴 **被阻塞**：必须等待**成员 A** 产出 `Data` 对象后，才能测试前向传播和训练循环。
    *   🟢 **可并行**：在等待 A 的过程中，可以先写好 `model.py` 的骨架代码以及报告中关于 GCN 原理的文字。
    *   🔴 **阻塞他人**：**必须与 A 联调跑通基线**，否则成员 C、D 的实验无代码可跑。

### **成员 C：损失函数优化 (解决不平衡)**：Zhou Haoxiang
*   **代码与实验任务**：在 `train.py` 中实现 Weighted CrossEntropy。跑出标准 CE 与 Weighted CE 对比图和前100个测试结果对比表。
*   **报告与答辩任务**：撰写报告的 *Design & Functions* (Loss设计与不平衡解决)。答辩时讲解权重 Loss 机制及优化后的对比结果。
*   **执行状态**：
    *   🔴 **被阻塞**：必须等待**成员 B** 提交能跑通的基线 `train.py`，才能在其基础上修改 Loss。
    *   🟢 **可直接执行**：等待期间可以先撰写报告中关于“为何选择 Weighted CE 解决不平衡”的理论段落。
    *   🟢 **完全并行**：一旦拿到基线代码，实验过程与 D、E **完全并行**，互不干扰，也无需将实验代码合回主分支。

### **成员 D：学习率调参与环境统筹** ：(YNC)
*   **代码与实验任务**：编写 `plot_metrics.py` 统一画图脚本；编写 LR 批量实验脚本。跑出不同 LR 0.1/0.01/0.001 的对比图。
*   **报告与答辩任务**：撰写报告的 *Demonstration & Performance* (学习率对收敛的影响)。答辩时分析不同学习率下的训练曲线与最优选择。兼任团队的**运维 (Ops)**，负责合并检查和一键回滚。
*   **执行状态**：
    *   🟢 **可直接执行**：`plot_metrics.py` 画图脚本可以立即编写，规定好团队统一的图表颜色、字体大小，并分发给所有人。
    *   🔴 **被阻塞**：LR 实验脚本必须等待**成员 B** 提交基线代码后才能运行。
    *   🟢 **完全并行**：拿到基线代码后，实验过程与 C、E **完全并行**。

### **成员 E：子图采样与 Batch Size 实验**：FanWenyan
*   **代码与实验任务**：引入 `NeighborLoader` 改造整图代码为 Batch 训练。跑出不同 Batch Size 的对比图。
*   **报告与答辩任务**：撰写报告的 *Demonstration & Performance* (Batch Size 与显存/稳定性分析)。答辩时讲解子图采样原理，展示不同 Batch Size 的结果。
*   **执行状态**：
    *   🔴 **被阻塞**：代码改造必须等待**成员 A** 产出 `Data` 对象，且最好基于**成员 B** 的基线代码进行修改。
    *   🟢 **可直接执行**：等待期间可以先学习 PyG 的 `NeighborLoader` 用法，并撰写报告中关于“整图 OOM 与采样原理”的理论段落。
    *   🟢 **完全并行**：拿到基线代码后，实验过程与 C、D **完全并行**。

### **成员 F：综合评估**：Li Ya
*   **代码与实验任务**：编写 `evaluate.py` (实现不平衡指标如 F1-Score, PR-AUC)。跑出最终的综合性能雷达图或柱状图。
*   **报告与答辩任务**：撰写报告的 *Conclusion* & *Future Work*。答辩时总结最终成绩单，分析局限性并主持 Q&A。兼任团队的**产品经理 (PM) 和 QA**，负责汇总全员报告与 PPT。
*   **执行状态**：
    *   🟢 **可直接执行**：`evaluate.py` 可以使用随机生成的 dummy 数据先行编写和测试逻辑，不强依赖真实数据。可以立即搭建 Word 报告的空壳模板和 PPT 母版。
    *   🔴 **被阻塞**：最终指标的计算需要等待**成员 C、D、E** 跑出最优参数组合后，运行一次最终模型。
